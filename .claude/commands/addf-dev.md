# addf-dev — 1タスク実施

## 手順

### 1. コンテキスト読み込み

以下を順番に読む:

1. `.claude/Feedback.md` — 未対応の改善アクションを確認
2. `TODO.md` — 未着手タスクを列挙
3. `.claude/Progress.md` — 進行中タスクがあれば継続、なければ次へ

### 2. タスク選択

TODO.md の「未着手」タスクを対象に、以下の優先順位で1つ選ぶ:

**優先1 — 複利効果**: 他タスクのブロッカー解消・インフラ整備・テスト基盤・ツール改善など、
実施することで後続タスクが楽になるもの。

**優先2 — 若番**: 依存関係がない場合、計画ファイルの番号が小さいもの
（例: `phase21.4` < `phase22.1`、`0026` < `0028`。命名はプロジェクトによる）。

選んだタスクを一行で示す: `→ <計画ファイル名> を実施します`

**アイドル時（選べるタスクがない場合）**: 未着手がない、または残る未着手が全て「要確認（質問投下済み）」
「オーナー指示待ち」のときは、`addf-Behavior.toml` の `[speculation].enable` が true なら
`/addf-speculate` を1サイクル実行して手順4（完了処理）へ進む。false なら従来どおりオーナーに確認する。

### 3. 実装

`docs/plans/` から選んだ計画ファイルを読み、CLAUDE.md のブートシーケンスに従って実装する
（`CLAUDE.repo.md` でプランディレクトリが追加定義されている場合はそちらも対象）。
`.claude/Progress.md` の運用ルールに従い、**2段階品質ゲート**（Stage 1: ビルド・Lint・テスト → Stage 2: quality-gate チーム）を完遂する。

**日記（代替わり引き継ぎ）**: サブタスク完了時・重要な判断の直後・計画変更時に、Progress.md のタスク「#### 日記」へ4項目エントリー（やったこと/今の見立て/次の自分へ/気になっていること）を書く。書式は Progress.md 運用ルール 3.5 参照。`/loop` の次イテレーションの自分が最初に読む引き継ぎ文書になる。

**迷ったとき（閾値割れ）**: CLAUDE.md「迷ったときの作法（7割共有原則）」に従う。

- 確信度が閾値（デフォルト7割）を下回り responsiveness が `relaxed` の場合: `.claude/Questions.md` に質問を置き、TODO の該当タスクを「要確認（質問投下済み）」にして**次のタスクを選び直す**（ステップ2へ戻る）
- `unattended` の場合: 質問を置いた上で最有力解釈により投機続行し、作業を `speculative/` ブランチに隔離する
- worktree 隔離下で実装している場合は閾値を1段下げてよい（失敗を捨てられるため）

### 4. 完了処理

`.claude/Progress.md` の運用ルールのうち「ノウハウ蓄積」「フィードバック記録」「アーカイブとコミット」（ステップ 9〜15）を全て実行し、コミットする。

### 5. ループ継続

`/loop` が次の実行を自動スケジュールするため、1サイクルの処理を終えたら完了とする。
次のサイクルで再びステップ1から再開される。

## 経験の活用
- 実行前に `addf-dev.exp.md` が存在すれば読み、過去の経験（タスク選択判断、よくあるトラブル等）を考慮する
- 実行後、新たな教訓があれば `addf-dev.exp.md` に追記する
