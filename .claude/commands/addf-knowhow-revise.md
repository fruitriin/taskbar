# addf-knowhow-revise — ノウハウの再検証・訂正

古いノウハウは静かに嘘になる。依存ライブラリの更新・設計の刷新・スキルの変更により「過去は正しかったが今は誤り」になった知見を、未来のエージェントが古文書として信じる事故を防ぐ。

## 引数

- **引数なし**: 🔴 stale と `needs-review` の全ファイルを対象にする
- **ファイルパス指定**: そのファイルだけを再検証する

## 手順

### 1. 対象の特定

1. `docs/knowhow/INDEX.addf.md`（ダウンストリームでは `INDEX.md`）を読み、🔴 stale / `needs-review` のファイルを列挙する
2. 引数でファイル指定があればそれを優先する

### 2. 再検証（1ファイルずつ）

3. 対象ノウハウを読み、主張・前提・依存（`depends_on`）を把握する
4. 依存先の現状を読み直す（参照されているファイル・スキル・ライブラリのバージョン等）
5. ノウハウの主張が現在も妥当か判定し、以下のいずれかを実行する:

| 判定 | アクション |
|---|---|
| 妥当 | フロントマターの `last_verified` を今日に更新するのみ |
| 部分的に誤り | 該当箇所を訂正し、末尾の「## 訂正履歴」に追記。`last_verified` 更新 |
| 後継が存在する | `status: superseded` に変更し、`superseded_by:` に後継パスを記載 |
| 後継なしで参照不要 | `status: retired` に変更（棚上げ。歴史として保管し、削除はしない） |

### 3. 訂正履歴の書式

```markdown
## 訂正履歴

### YYYY-MM-DD
- <何が誤っていたか> → <何に訂正したか>（根拠: <確認したソース>）
```

### 4. 完了処理

6. 全対象の処理後、`/addf-knowhow-index reindex` を実行して INDEX に反映する
7. 訂正・遷移の件数を報告する

## 語彙の注意

- **`deprecated` は使わない**。ノウハウは過去のエージェント（≒ 同僚）が残した仕事であり、烙印ではなく敬意をもって `superseded`（次の世代に引き継がれた）/ `retired`（現役を退いた）と呼ぶ
- retired にしても削除はしない。📜 棚に置かれた歴史として保管する

## 経験の活用

- 実行前に `addf-knowhow-revise.exp.md` が存在すれば読み、過去の経験を考慮する
- 実行後、新たな教訓があれば `addf-knowhow-revise.exp.md` に追記する
