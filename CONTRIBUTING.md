# コントリビューションガイド

[English CONTRIBUTING](CONTRIBUTING.en.md)

## 基本方針

このプロジェクトは **計画駆動（Plan-driven）** のコントリビューションモデルを採用しています。

```
コードではなく計画をレビューする。
筋の良い計画は受け入れ、実装はAIが担保する。
```

---

## コントリビューションの流れ

### 1. Issue / PR の先頭に「きっかけ」を書く

すべてのIssue・PRには、改善のきっかけとなった情報を先頭に含めてください:

```markdown
## きっかけ
- 元となったプロンプト/会話: （概要またはリンク）
- アイデアの出典: （ブログ記事、他プロジェクト、自分の着想など）
- 関連Issue: #XX
```

これはコードの帰属ではなく **アイデアの系譜を追跡する** ためです。

### 2. 計画ファイル（Plan）を含める

実装を伴うPRには、計画ファイルを含めてください:

```
docs/plans/NNNN-タイトル.md
```

計画ファイルには以下を記述:

```markdown
# 計画: タイトル

## 動機
なぜこの変更が必要か

## 設計
どう実装するか（CLAUDE.mdの方針に沿っているか）

## 影響範囲
どのモジュール・ファイルに影響するか
```

### 3. レビュープロセス

```
計画の筋 → 人間が判断（Accept / Reject / 要議論）
コードの詳細 → CLAUDE.md + CI が担保
```

- **人間がレビューするもの**: 計画の方向性、アーキテクチャ上の判断
- **人間がレビューしないもの**: コードのスタイル、実装の詳細、リファクタリングの手法
- **AIが担保するもの**: CLAUDE.mdへの準拠、禁止事項の遵守、コード品質

---

## CLAUDE.md を改善した場合

CLAUDE.mdはプロジェクトの憲法です。改善提案は歓迎しますが:

1. まずIssueで提案内容を説明してください
2. オーナーの承認を得てからPRを作成してください
3. CLAUDE.mdの変更は必ず人間がレビューします

---

## やらないこと

- コードの行単位でのスタイルレビュー
- 「このメソッド名はこうすべき」等の些末な議論
- 計画なしのコードだけのPR（バグ修正の1行パッチは除く）
