# 開発プロセス

```
計画（Plan）→ 実装 → 品質検証 → コミット
```

## 基本原則

- **計画駆動**: コードではなく計画をレビューする。筋の良い計画は受け入れ、実装は AI が担保する
- **ブートシーケンス**: セッション開始時に Feedback → TODO → Progress を順に読み込み、現在の状態を把握
- **品質ゲート**: ビルド・Lint・テスト → コードレビュー → セキュリティレビュー（オプション）
- **並列実装**: git worktree を活用したサブタスクの並列実行

## ブートシーケンス

セッション開始時に自動実行される手順（正は CLAUDE.md。ここは概要のみ）:

1. `.claude/Feedback.md` を読む — 未対応の改善アクションを確認
   - 1.5. `.claude/Questions.md` を読む — オーナーの回答があれば Plan に反映
   - 1.6. `.claude/Dashboard.md` があれば提示 — unattended 自走の差分まとめ
2. `TODO.md` を読む — タスクバックログと優先度を把握
3. `.claude/Progress.md` を読む — 進行中タスクがあれば、日記の末尾3エントリーで前任者の文脈を引き継いで継続
4. TODO に未完了タスクがない場合 — オーナーに次のタスクを確認（プロジェクト初回は骨格プランニング）
5. knowhow サブエージェントを起動 — Plan に関連するノウハウをフィルタリング

## タスクのライフサイクル

### 開始時
- Feedback.md の改善アクションを確認
- サブタスクをチェックリストとして `.claude/Progress.md` に記録
- 並行作業できる粒度に分割

### 実装中
- サブタスク完了時にチェック
- 並列可能なタスクは git worktree で分離
- サブタスク完了時・重要判断の直後に **日記** を書く（下記）

## 代替わり日記

エージェントは resume・compaction・`/loop` の次イテレーションのたびに「小さな代替わり」をしている。
次の代の自分は**同僚でもあり、寝て起きたあとの自分でもある**。同じモデルなら性格・スキル・考え方の癖は重みとして引き継がれていて、失われるのはエピソード記憶だけ — このとき日記は初見の引き継ぎ書ではなく、「あー、そうそう、こんな感じ」と思い出すための手がかりとして働く。モデルが変わるときは本当の代替わりとなり、日記は同僚への手紙になる。自分宛てに誠実に書いたメモは同僚にも読める（逆は成り立たない）ため、同じ書式が両方に効く。Progress.md の「日記」セクションで状況を引き継ぐ。

- **書式**: 4項目（やったこと / 今の見立て / 次の自分へ / 気になっていること）
- **タイミング**: サブタスク完了時・重要な判断の直後・計画変更時・compaction を予感したとき
- **分業**: ブランチ checkpoint = 何がコミットされたか（事実）、日記 = なぜそうしたか（文脈）。両方で前任者の靴に履き替えられる
- **自動化しない**: 日記を自動生成するフックは意図的に導入していない。書く行為自体が思考の整理であり、次の自分への手紙として人格を持って書く
- タスク完了時は日記ごと `.claude/Progresses/` にアーカイブされ、失敗・想定外の判断は後日 knowhow 化の原資料になる

「遺書」ではなく「日記」という語彙を使う。死を意識した記録は書く側の温度を下げる。
人格の連続性を仮定して書くことで、読み手（次の代の自分）も自然に状況へ入れる。

### 完了時（品質検証）
1. ビルド・Lint・テスト実行
2. コードレビュー（`addf-code-review-agent`）
3. コントリビューション検出（`addf-contribution-agent`）
4. レビュー指摘への対応（Critical/High は必須修正）
5. 品質ゲート知見・タスク総括を `/addf-knowhow` で記録
6. Plan に実装結果を反映
7. Feedback.md に問題・改善アクションを記録
8. Progress.md を日記ごとアーカイブ
9. コミット

（正確なステップ番号と詳細は `.claude/templates/ProgressTemplate.addf.md` の運用ルールを参照）

## 自律開発ループ

```
/loop 1h /addf-dev
```

AI エージェントが自動でタスクを選択し、上記のライフサイクルを繰り返します。
タスク選択の優先順位: ブロッカー解消 > インフラ整備 > 若番。

詳細は [CONTRIBUTING.md](../../CONTRIBUTING.md) を参照してください。
