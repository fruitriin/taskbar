# electron-store のマイグレーションと defaults の落とし穴

**記録日**: 2026-07-05（app-window-grouping 実装時）

## 1. マイグレーションは package.json の version で発火判定される

electron-store（内部の conf）のマイグレーションキーは
`semver.satisfies(現在のアプリバージョン, キー)` で評価される。

- 例: アプリの version が `2.1.0` のとき、`'>=2.1.1'` キーのマイグレーションは**実行されない**
- 「マイグレーションを書いたのにユーザー環境で発火しない」場合、まずバージョンとキーの関係を疑う
- 新フィールド追加のマイグレーションを書いたら、**リリース時に version をキーが満たされる値まで上げる**ことをリリースチェックに含める

## 2. defaults はトップレベルキー単位の浅いマージ

`defaults.options.newField` を追加しても、既存ユーザーのストアに `options` キーが
既に存在する場合、`newField` は補われない（options オブジェクト全体が保存値で置き換わるため）。

対策は3点セット:
1. defaults に追加（新規インストール用）
2. マイグレーションで補完（既存ユーザー用。ただし上記1の発火条件に注意）
3. 読み出し側で `?? デフォルト値` フォールバック（バージョンアップまでの橋渡し・自己修復）

taskbar での実例: `options.appOrder` 追加時、renderer 側は `this.options?.appOrder ?? []` で読む。

## 3. マイグレーションのテスト

`src/main/tests/funcs/store-migration.test.ts` の慣習: マイグレーション関数と同じロジックを
テスト内に再現し、モックストアに対して「旧形状 → 新形状」を検証する。
新しいマイグレーションを書いたらテストケースを追加する（`appOrder migration (>=2.1.1)` 参照）。
