# Tauri コマンドの信頼境界と native メニューの並行性（Phase 3.5 レビュー知見）

**記録日**: 2026-07-14（rearch Phase 3.5 ペルソナ並列レビューの総括）

## Tauri コマンドの信頼境界

- **コマンド引数は webview JS が自由に組み立てられる**。capabilities の ACL は「どのコマンドを
  呼べるか」を制限するだけで、引数の値は検証しない。危険な副作用（kill・ファイル操作・
  ウィンドウ操作）を持つコマンドは、クライアント供給の構造体をそのまま使わず、
  **サーバー側の権威データと突き合わせて差し替える**（例: context_task は受け取った
  MacWindow を (owner_pid, window_number) で現在の CGWindowList と照合し、一致した
  サーバー側エントリをメニュー対象にする）
- 逆に `window: tauri::WebviewWindow` 型の引数は **Tauri が呼び出し元を自動注入**するため
  JS から偽装できない。呼び出し元の識別はこれを使う
- 防御は設定に依存させない: 「自ウィンドウはデフォルトフィルタで一覧に出ないから安全」は
  ユーザーがフィルタを変更できる時点で成立しない。最終防御（自プロセス pid 拒否など）は
  副作用の直前に置く

## native メニュー（tauri::menu）の並行性

- `run_on_main_thread` のクロージャは kCFRunLoopCommonModes で登録されるため、
  **popup の NSMenu トラッキングループ中にも後続クロージャが実行される**。
  「メニュー表示中に次の表示要求は来ない」は成立しない前提。グローバル状態
  （対象ウィンドウ等）を Mutex で受け渡すなら **single-flight ガード**（表示中フラグ）と
  **take() による消費**をセットで入れる
- `AppHandle::on_menu_event` は**追記型**（Vec への push）。popup のたびに登録すると
  ハンドラが増殖する。setup で1回だけ登録し、メニュー ID に接頭辞を付けて分岐する
- popup はメニューが閉じるまでブロックし、項目選択イベントは popup 復帰**後**に配送される。
  「popup 復帰直後に状態をクリア」するとイベントハンドラが対象を見失うので、
  クリアはハンドラ側の take() に任せる

## 座標計算のクランプ非対称に注意

- 位置計算関数を複数持つ場合（カーソル基準／ウィンドウ基準）、片方だけにモニタ内クランプが
  あると、もう片方が負座標・狭小モニタ構成で画面外に出る。**フォールバックは I/O エラーで
  しか発動しない**（値が変でも Ok が返る）ため、クランプは各計算パスに入れる

## プロセス面（ペルソナ並列レビューの効用）

- 委譲実装＋全ゲート緑でも Critical は残る（今回: 信頼境界の無検証、レース）。
  ゲートは「書いたものが正しく動く」を保証するが「書くべきでないものを書いた」は検出しない。
  マイルストーンのペルソナ並列（skeptic/attacker/newcomer）は、attacker が信頼境界、
  skeptic が並行性前提、newcomer がドキュメントと実体のズレ、と**きれいに非重複の指摘**を
  出した。2ペルソナ重複指摘（レース・クランプ）の格上げ運用も機能した
- レビューへの依頼文に「tauri のソース（~/.cargo/registry）で裏取りしてよい」ことが伝わる
  情報（バージョン・環境）を含めると、指摘が推測でなくソース裏取り付きで返ってくる
