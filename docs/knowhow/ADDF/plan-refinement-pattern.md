# 粗々プランの詰め方 — 未決事項を「決定＋根拠」に1:1変換する

> 出典: Plan 0028 の詳細化タスク（2026-07-03）。「粗々の起票」を実装可能な計画へ改訂した際の作法

## 発見した知見

### 詰めの本体は「未決事項 → 決定」の1:1変換

粗々プラン（0028/0029 形式）の価値は未決事項が列挙されていること。詰めるときは各未決を
**「決定N: <結論> + 根拠」のセクションに1:1で変換**し、旧「未決事項」節は消す。
決められないものだけ「残る未決事項」に残し、**なぜ今決めないか**（オーナー判断待ち / 実装時の
実地確認が必要 / YAGNI）を必ず添える。「検討する」のまま残る項目が多いほど詰めは失敗している。

### 決定の材料は knowhow フィルタ + 実装面の実地確認の両輪

- knowhow フィルタ（addf-knowhow-agent）で設計原則側の材料を集める
- 並行して、計画が触れる**実物**（設定ファイルのスキーマ・スキル本文・lint スクリプトの実際の検証範囲・
  gitignore ブロック）を自分で読む。「lint-toml は構文チェックのみ」のような事実は knowhow ではなく
  実物にしか書かれていない
- 引用は本文と突合してから書く（レビューエージェントが全引用を実物確認するため、推測引用はここで露見する）

### 詰め後のレビューは「決定間の整合」を突かせると効く

実装レビューと違い、計画レビューの主戦場は**決定同士の矛盾**。Plan 0028 では単体では正しい決定が
組み合わせで3件の Medium を生んだ:

1. **報告先の重複規定**: 2つの節が同じイベント（テスト失敗・衝突）の報告先を別々に定めていた
   → 既存書式（Dashboard の2セクション）の役割に合わせて一本化
2. **「再構築できる」の過大表現**: 「git が真実源」と言ったとき、git から再構築できるのは「存在」だけで
   状態メタデータ（テスト通過/失敗・出典・更新時刻）は復元できない → 再構築の保証範囲を明示し、
   復元エントリの初期化規則（状態「要再検証」）を定義
3. **既存フローとの接続未定義**: 新分岐が既存スキルのどのステップに合流するか（完了処理を経由するか）を
   書き忘れる → gitignore ファイルにしか痕跡が残らない実行経路は「silent に捨てない」違反になりうる

レビュー依頼時に「決定N同士・旧版から引き継いだ記述との矛盾」を観点として明示すると検出率が上がる。

### 完了条件はA型/B型を一文に混在させない

「テストが PASS（A型）＋ Dashboard に記録される（B型・エージェント自然文生成）」を一文に書くと、
テスト通過で B型要件も満たしたと誤認される。機械検証できる範囲を明記し、
B型部分は `<!-- human-judgment -->` で分離する（Plan 0027 のドクトリンを計画文書にも適用）。

## 注意点・制約

- この作法は「粗々の起票」形式（設計の方向性と未決を出す段階を明示したプラン）が前提。
  未決が列挙されていない旧形式プランでは、先に未決の洗い出しから始める
- 詰めても「実装状況: 未着手」のまま（詳細化は実装ではない）。ヘッダに詳細化日を添えると
  着手時に鮮度が判断できる
