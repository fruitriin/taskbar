# Plan 状態の信用ベース運用 — 疑う仕事は lint に任せ、表は基本信用する

> 出典: Plan 0015 の完了確認（2026-06-10）と Plan 0024 の機械化（2026-06-11）。
> 初版は「『未着手』表記は信用せず git log で検証する」だったが、
> オーナーの設計方針（疑いは一瞬・基本は信用）に基づき lint 化とセットで改訂した。

## 運用ルール

- **TODO の状態表記は基本的に信用してよい。** lint ペア6
  （`lint-template-sync.py`、TODO 状態 ⇔ Plan `## 実装状況:` ヘッダの突合）が
  品質ゲートの `run-all.sh` 経由で毎タスク完了時に走り、状態の矛盾・参照切れ・
  登録漏れを WARNING で検出する。読む側が着手のたびに疑うコストは払わない
- **疑うのは次の2つの場面だけ**:
  1. lint がペア6の WARNING を出したとき — 完了処理の反映漏れを疑い、
     `git log --oneline -- <関連ファイル>` で実態を確認して同期する（突合は1分で安い）
  2. 長期間「未着手」のまま滞留している Plan を拾うとき — lint は
     `## 実装状況:` ヘッダの無い旧 Plan を検査しない（寛容設計）ため、
     ヘッダの無い古い Plan に着手する場合のみ着手前に git log で実態と突合する
- 新しく起案する Plan には `## 実装状況:` ヘッダを必ず付ける。
  これがあれば lint の保護対象に入り、以後は信用ベースに乗る

## 遡及付与 — 疑いの一括清算（ペア6導入時に推奨）

ヘッダ無し旧 Plan が残っていると「着手時に疑う」例外路が残り続ける。
ペア6導入時に旧 Plan へヘッダを遡及付与すると、例外路を通る機会がほぼ消滅する。

- **TODO の状態を盲目的に転記しないこと**。TODO 自体の誤りが警戒対象なので、
  無検証の転記は誤りを Plan 側にロンダリングし、lint が偽の一致にお墨付きを与える
- 手順: 各 Plan について (1) アーカイブ済み Progress・完了コミット（git log）・
  Plan 本文を1回だけ突合 → (2) `## 実装状況: 完了（<完了日>、<実施日> 遡及確認）` を
  タイトル直後に付与。1 Plan あたり1〜2分
- 「最後に一回だけ全部疑い、以後は二度と疑わない」一括清算。ADDF 本体では
  2026-06-11 に 20 本へ実施した（完了日はアーカイブ済み Progress の日付を採用）

## 背景（なぜドリフトが起きたか）

- Plan 0015 は計画ファイル追加と同一コミットで実装まで完了していたが、
  TODO.addf.md が「未着手」のまま3ヶ月放置され、後続エージェントは皆それを信じた
- 計画と実装が同一コミットに入ると「実装も済んだ」ことが TODO 更新漏れで見えなくなる。
  完了処理（TODO 反映）が漏れたときの典型パターン
- 対処の変遷: 初版はこの knowhow で「毎回疑え」と読む側に恒常コストを課したが、
  Feedback.md の原理「意思で覚えず機械化する」に従い Plan 0024 で lint 化。
  書く側の規律を機械が検査することで、読む側は信用ベースに戻れた

## プロジェクトへの適用

- 実装済みと判明した Plan は: 実装状況セクションを追記して完了化し、
  **現状との差分（鮮度ドリフト等）だけを新 Plan に切り出す**（Plan 0015 → 0022 の実例）
- 計画と実装を同一コミットにする場合は、同じコミットで TODO の状態も「完了」にする
  （Plan 0024 自身がこのパターンの実例）

## 注意点・制約

- 「実装済みに見えるが計画の一部だけ」のケースがある。完了化する前に
  Plan の設計項目を1つずつ現物と突合すること
- lint が flag するのは完了⇔未着手の明確な矛盾のみ。「進行中」等の中間状態は
  誤検出回避のため対象外（信用ベースの精神: 誤検出で疑心を増やすより見逃しを許容する）

## 訂正履歴

- 2026-06-11: 「着手前に毎回 git log で疑う」→「lint ペア6を前提とした信用ベース」へ
  全面改訂（Plan 0024）。旧運用は lint 不在時の暫定として正しかったが、恒常コストが
  オーナーの設計方針と逆向きだった
