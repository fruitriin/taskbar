# マルチディスプレイ構成でのレンダラーローカル状態の罠

**記録日**: 2026-07-05（app-window-grouping 実装時）

## 前提

taskbar.fm はディスプレイごとに独立した BrowserWindow（＝独立した Vue インスタンス）を作る。
メインプロセスは同一の `macWindowProcesses` を全タスクバーに配信する（helper.ts）。

## 罠

Vue コンポーネントの `data()` に持たせた状態（例: 出現順リスト、ドラッグ中フラグ）は
**ディスプレイごとに別物**になる。さらにディスプレイの追加・削除・メトリクス変更で
タスクバーウィンドウは再作成され、そのレンダラーのローカル状態はリセットされる。

- 「セッション内で安定」のつもりのローカル状態は、ディスプレイ再構成のたびに消える
- ウィンドウをまたぐ HTML5 D&D では、dragstart 側の `this` の状態は drop 側から見えない

## 対策パターン

1. **決定的ルールに落とす**: 状態に頼らず、入力データから常に同じ結果が出るソートキーを使う
   （例: グループ順のフォールバックを「グループ内最小 kCGWindowNumber 昇順」にすると
   全レンダラーが同じ順序に収束し、リセットされても揺れない）
2. **dataTransfer で運ぶ**: クロスウィンドウ D&D は `event.dataTransfer.setData/getData` で
   識別子を運ぶ（Electron の BrowserWindow 間でも機能する）。レンダラーローカルの
   `dragApp` はウィンドウ内 D&D 用のフォールバックとして残す
3. **本当に共有が必要なら electron-store + updateOptions 配信**: ユーザー操作で確定した
   状態（appOrder 等）はメインプロセス経由で永続化し、全レンダラーに配信する
