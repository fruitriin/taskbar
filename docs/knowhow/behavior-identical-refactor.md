# 挙動同一リファクタの運用パターン

**記録日**: 2026-07-06（リアーキ Phase 1 スライス 1-B〜1-D の総括）

Options API → Composition API 変換やエントリー統合のような「挙動を変えないことが
正しさの基準」のリファクタで効いた運用。

## 効いたこと

1. **レビュー指示に比較基準のコミットハッシュを明記する**
   「git show <hash>:<path> と項目別に差分照合せよ」と指示すると、レビューが
   機械的に等価性を検証できる。1-C 後半ではテンプレート参照識別子22個の網羅・
   IPC 引数の等価性まで照合された。曖昧に「挙動同一か見て」では出ない精度
2. **意図的な逸脱はコミットメッセージに列挙する**
   死にコード削除・console.log 除去・removeAllListeners → ペア解除などの「ついで改善」は
   挙動同一原則からの逸脱。分離コミットにするか、最低限メッセージに明記しないと
   「diff ゼロ期待」のレビュー・CI と衝突する
3. **N ファイルを1つに統合するときは全ペアの diff を取る**
   1-B で option.html と index.html の diff だけ見て「script src 以外同一」と誤判断し、
   menu/fullWindowList.html 固有の Font Awesome CDN link・CSP・フォント指定を見落とした
   （レビューが Critical で検出）。統合対象全体の union 比較が必須
4. **レビューの「事実誤認の検出」を歓迎する**
   1-D では自分の「echo ループになる」という設計根拠が、実配線の検証で誤りと判明した
   （optionWindow は updateOptions の配信対象外）。設計判断の根拠はコメントに書き、
   レビューに「この分析を独立に検証せよ」と明示的に依頼すると誤ったメンタルモデルの
   固定化を防げる

## 落とし穴

- **フォーマッタは「挙動同一」を破ることがある**: prettier がテンプレート内インライン式の
  セミコロンを剥がすと vite:vue の式パースが壊れた（実測）。複数文のインラインハンドラは
  関数に抽出しておくのが恒久対策
- **Options API では黙認されていた死にコード**（未使用 data/methods）は script setup ＋
  noUnusedLocals で即エラーになる。変換前に消すか、変換時に「意図的逸脱」として扱う
